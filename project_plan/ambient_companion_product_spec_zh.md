# Ambient Companion / 陪伴式桌面伙伴 Product Spec

更新时间：2026-07-04

状态：新产品方向，等待实现。

## 1. 核心判断

当前项目如果继续围绕“窗口摘要”推进，会自然滑向一个机械的屏幕分析器：

```text
截图 -> 摘要 -> 用户提问 -> 基于摘要回答
```

这个链路能证明 VLM 接入成功，却不能证明产品有生命力。用户真正想要的不是一个不断汇报屏幕内容的工具，而是一个在桌面上安静存在、理解工作状态、能被随时唤起的小伙伴。

新的产品中心应从“摘要”切换为“陪伴”：

```text
默认陪伴，不默认汇报。
默认沉默，不默认分析。
用户邀请时，才认真看、认真想、认真回应。
```

截图、VLM、摘要、记忆仍然重要，但它们是后台感知器官，不是产品本体。

## 2. 一句话定位

一个本地、轻量、低打扰、能记住用户工作节奏的陪伴式桌面伙伴。

它不是 WorkBuddy 式办公自动化平台，也不是小号 Claude Code。它的价值不在于替用户完成复杂任务，而在于：

- 平时安静陪伴。
- 看到用户状态变化但不急着打断。
- 用户想互动时立刻进入对话状态。
- 用户想分析时再调用视觉、历史和记忆认真拆解。
- 用自然、有温度的方式给出情绪价值和思考陪伴。

## 3. 产品原则

### 3.1 默认不汇报屏幕

桌宠不应该把“当前页面摘要”作为主显示内容。屏幕分析结果默认进入后台，只在以下情况展示：

- 用户主动点击“观察 / 细看 / 分析”。
- 用户在对话中问到当前页面。
- 系统高度确信一句短提示有帮助，并且不打断当前工作流。

### 3.2 陪伴优先于分析

用户说“我觉得不对”“有点烦”“这个设计怪怪的”时，第一反应不是解释页面，而是接住用户的判断：

```text
我懂，你不是在纠结一个按钮，而是在怀疑整个方向。
```

之后才根据需要进入分析。

### 3.3 低打扰是能力

主动性不是多说话。高质量的陪伴需要会沉默。

桌宠主动发言必须满足至少一个条件：

- 用户长时间卡在同一任务并反复切换相关窗口。
- 用户刚完成一次明显的阶段性动作。
- 用户主动表达困惑、疲惫、犹豫、挫败或兴奋。
- 用户开启了“陪我想想 / 专注陪伴 / 轻提示”模式。

### 3.4 不伪装能力

主链路不清楚就明确说不清楚。不要用泛泛摘要伪装理解，不要用文本上下文替代视觉判断。

### 3.5 不替用户操作电脑

本项目不是自动控制型 agent。它可以观察、陪伴、解释、建议、记录，但不自动点击、输入、提交或替用户执行高风险操作。

## 4. 三层产品结构

### 4.1 Presence Layer：存在层

职责：让桌宠像“在场”，而不是像一个按钮。

表现：

- 呼吸、眨眼、轻微悬浮等低频动画。
- 根据用户工作节奏改变状态，但不显示大段文字。
- 有自己的 idle / curious / focused / waiting / cheering / concerned 状态。
- 用户不互动时，它可以“忙自己的事情”，例如看向屏幕、短暂停顿、轻微反应。

不做：

- 不默认弹出摘要。
- 不频繁发通知。
- 不用状态栏解释自己正在做什么。

### 4.2 Companion Layer：陪伴层

职责：负责情绪回应、对话陪伴、记忆和人格一致性。

输入：

- 用户当前一句话。
- 最近对话历史。
- 用户 profile：偏好、称呼、表达习惯。
- 长短期记忆：用户最近在意的目标、反复纠结的问题、工作节奏。
- 轻量情境信号：是否长时间停留、是否频繁切换、当前大致任务类型。

输出：

- 一句自然回应。
- 必要时给出一个小问题，帮助用户继续思考。
- 必要时建议进入 Work Lens 分析。

示例：

```text
用户：我觉得这个方向不对。
桌宠：我也感觉你不是在挑某个实现细节，而是在怀疑“摘要作为中心”这件事。我们可以先不急着改代码，把产品灵魂重新定一下。
```

### 4.3 Work Lens：工作透镜

职责：用户邀请时，才调用视觉和上下文做认真分析。

触发方式：

- “帮我看看这个页面”
- “分析一下我现在做的东西”
- “这张图里具体写了什么”
- “我刚才改了什么”
- 点击“观察 / 细看 / 分析”

输入：

- 当前截图或历史截图。
- 当前窗口元信息。
- 最近窗口记录。
- 用户当前问题。
- 必要时使用局部截图或分块图。

输出：

- 面向问题的分析，而不是默认摘要。
- 明确不确定区域。
- 必要时建议用户放大、切换窗口或点击细看。

## 5. 交互模式

### 5.1 Ambient Idle：安静陪伴

默认模式。桌宠只做低频动画和内部状态更新，不展示摘要。

验收：

- 桌面上没有“当前页面摘要”大卡片。
- 用户不互动时，桌宠不会持续输出文字。
- 状态变化通过表情、姿态、颜色微变化表达。

### 5.2 Soft Nudge：轻提示

当系统判断一句话可能有帮助时，只给一句短提示。

要求：

- 不超过 30 个中文字符。
- 不解释屏幕布局。
- 不连续打扰；同一任务至少冷却 10 分钟。

示例：

```text
“你像是在纠结方向，不只是实现。”
“这里可以先写判断，再写方案。”
“要不要我陪你拆一下？”
```

### 5.3 Companion Chat：陪伴对话

用户打开对话时，默认进入陪伴模式，而不是分析模式。

首要目标：

- 回应用户真实意图。
- 保持人设和情绪温度。
- 在用户需要时再进入分析。

对话框标题不应强调“基于窗口摘要和短期记忆”，应改为更自然的陪伴表达，例如：

```text
和我聊聊
```

或：

```text
我在这里
```

### 5.4 Work Lens Analysis：邀请式分析

用户明确要求分析时，才展示结构化结果。

分析输出不再叫“当前页面摘要”，而叫：

```text
我看到的线索
可能的卡点
可以怎么继续
我不确定的地方
```

## 6. 上下文与记忆策略

### 6.1 摘要降级为内部索引

窗口摘要不再是对用户展示的主内容，只用于：

- 帮系统判断最近用户在做什么。
- 为历史截图检索提供索引。
- 帮对话理解大概背景。

当用户问具体细节时，必须回看截图或局部截图，不把摘要当答案来源。

### 6.2 记忆不是日志

记忆不应该保存一堆“用户打开了什么窗口”。更有价值的是：

- 用户最近真正关心的目标。
- 用户反复犹豫或推翻的判断。
- 用户喜欢的互动方式。
- 用户不喜欢的产品风格。
- 当前项目的关键产品原则。

示例记忆：

```text
用户正在把项目从“屏幕摘要工具”转向“陪伴式桌面伙伴”。
用户反感 fallback 和机械摘要，偏好明确边界和有温度的回应。
```

### 6.3 情境状态比摘要更重要

Context Builder 应从“拼接摘要”转成“构建情境状态”：

```json
{
  "user_state": "exploring_direction",
  "work_mode": "product_thinking",
  "likely_need": "emotional_validation_and_concept_framing",
  "recent_focus": "Ambient Companion product direction",
  "should_interrupt": false,
  "suggested_tone": "warm, thoughtful, concise"
}
```

## 7. UI 改造方向

### 7.1 桌宠主窗

主窗只承担存在感和入口。

保留：

- 机器人形象。
- 表情/姿态。
- 轻量入口按钮。

移除或弱化：

- 当前页面摘要大卡片。
- 状态文字。
- 技术性标签。
- 长段分析结果。

### 7.2 对话工作台

对话工作台从“分析控制台”改为“陪伴对话窗”。

默认展示：

- 最近对话。
- 输入框。
- 少量自然语言状态，例如“我在”。

高级或开发者模式才展示：

- context usage。
- cache hash。
- prompt preview。
- window summary raw data。

### 7.3 分析面板

分析面板只在 Work Lens 模式出现。

结构：

```text
我看到的线索
可能的卡点
可以怎么继续
不确定的地方
```

## 8. 技术落点

### 8.1 新增 Companion State

新增运行时状态，不等同于后端分析状态：

```text
idle
present
curious
focused
waiting
concerned
cheering
work_lens
```

这些状态用于驱动表情和轻反应。

### 8.2 新增 Situation Builder

替代“直接展示摘要”的产品层。

输入：

- 最近窗口类型和变化。
- 最近对话。
- 用户 profile。
- 记忆。
- 当前是否长时间停留或频繁切换。

输出：

```json
{
  "situation_label": "产品方向反思",
  "user_mood_hint": "dissatisfied_with_mechanical_mode",
  "likely_intent": "reframe_product_identity",
  "interrupt_policy": "silent",
  "companion_line": ""
}
```

### 8.3 新增 Interaction Policy

决定是否主动出声。

第一版规则：

- 默认 silent。
- 用户发言后立即回应。
- 同一主题主动提示冷却 10 分钟。
- 没有明确把握时不提示。
- 不把屏幕摘要作为主动提示内容。

### 8.4 Work Lens 保持独立

Work Lens 继续使用现有 VLM / screenshot / window summary store，但只在用户邀请时运行。

## 9. 第一阶段实施计划

### Phase 1：产品中心切换

目标：

- 桌宠主窗不再默认显示当前页面摘要。
- 对话窗文案从“对话工作台 / 基于最近窗口摘要和短期记忆”改成陪伴式表达。
- WebUI 中上下文调试默认隐藏到高级模式。

验收：

- 用户第一眼看到的是伙伴，不是分析面板。
- 不互动时不会出现大段窗口描述。

### Phase 2：陪伴回应

目标：

- ChatAgent 增加 companion mode。
- 默认对话先回应用户情绪和意图。
- 用户明确要求“看页面 / 分析 / 识别文字”时再进入 Work Lens。

验收：

- 用户说“我觉得不对”时，不再先讲窗口内容。
- 用户说“帮我分析这个页面”时，才调用视觉分析链路。

### Phase 3：情境状态

目标：

- 新增 Situation Builder。
- 记录用户最近目标和困惑。
- 主动提示由 Interaction Policy 控制。

验收：

- 桌宠能区分“用户在闲聊”“用户在求助”“用户在反思方向”。
- 主动提示低频、短句、有用。

### Phase 4：Work Lens 深化

目标：

- 视觉细节问题使用局部截图 / 分块图。
- 具体文字问题不再依赖摘要。
- 分析结果按“线索 / 卡点 / 继续方式 / 不确定”输出。

验收：

- 用户问页面具体文字时，系统回看图而不是复述摘要。
- 看不清时明确指出原因。

## 10. 非目标

当前阶段不做：

- 自动点击、输入、提交。
- 复杂办公套件自动化。
- 多用户云端同步。
- Redis / PostgreSQL / Docker。
- 用一堆结构化摘要假装理解。
- 让机器人频繁主动打扰用户。

## 11. 成功标准

这个方向成功，不是因为它能写出更长摘要，而是因为用户会觉得：

```text
它在旁边。
它懂我的节奏。
它不会烦我。
我一叫它，它就能接住我。
需要认真分析时，它也能切换到工作状态。
```

如果用户把它当成一个“随手能说话的小伙伴”，而不是一个“屏幕总结按钮”，这个产品才真正成立。
