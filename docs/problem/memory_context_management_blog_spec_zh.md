# 个人网站博客 Spec：桌面伙伴的记忆与上下文管理问题

更新时间：2026-07-04

状态：给 WorkBuddy 扩写成文章的写作说明。

目标文件用途：这不是工程实现 spec，而是一份博客写作 brief。文章应从真实踩坑出发，讲清楚为什么桌面 VLM/Agent 项目的难点不在“能不能截图”，而在“如何把看到的东西、用户正在想的事、历史对话和长期偏好组织成干净上下文”。

## 1. 这件事有没有写作价值

有，而且价值不小。

它的价值不在于展示一个已经成熟的产品，而在于记录一个原型如何暴露出桌面 Agent 的核心矛盾：

- VLM 能看图，但“看见”不等于“理解用户此刻的问题”。
- 摘要能压缩屏幕，但摘要一旦成为答案来源，就会丢掉细节。
- 记忆能保存历史，但日志堆积不是记忆，反而会污染回答。
- 工具层能增强模型能力，但如果工具规划死板，用户会感觉系统很笨。
- 桌面伙伴需要陪伴感，但机械摘要会让它像一个客服窗口。

这是一篇适合个人网站的文章，因为它能体现作者的工程判断：不是堆技术名词，而是在原型失败中逐步找到产品边界和系统架构。

## 2. 推荐文章定位

建议标题：

- 《为什么“看见屏幕”还不够：桌面 AI 伙伴的上下文管理踩坑》
- 《从窗口摘要到陪伴式桌宠：一次桌面 Agent 原型的失败复盘》
- 《小模型桌面伙伴里的记忆问题：摘要、截图、工具调用和上下文断线》

一句话主旨：

> 桌面 AI 伙伴真正困难的不是调用 VLM，也不是保存历史，而是让模型在正确时刻拿到正确上下文，并且承认自己看不清、记不住、没证据。

文章风格：

- 第一人称工程复盘。
- 不包装成成功案例。
- 不把 WorkBuddy、Hermes、Claude Code 写成简单竞品比较，而是写成启发来源。
- 用“我以为 / 实际发现 / 后来调整”的结构让读者感到真实。

## 3. 核心问题清单

### 3.1 摘要陷阱：摘要不是理解

原型早期链路大概是：

```text
截图 -> VLM 生成窗口摘要 -> 用户提问 -> 基于摘要回答
```

问题：

- 摘要适合做索引，不适合回答细节问题。
- 用户问“页面里有什么好看的视频”“中间代码具体是什么”时，摘要通常只剩下概括，无法支持具体判断。
- 摘要越短，回答越像模板；摘要越长，又会污染上下文和响应速度。

文章要点：

- 摘要应该降级为 RAG 索引，而不是最终证据。
- 具体问题必须回看截图或局部截图。
- “看不清”比泛泛回答更诚实。

### 3.2 VLM 可见度问题：模型不是人眼

遇到的现象：

- 页面明明在人眼看来很清楚，VLM 却说不出具体文字。
- 有时文件夹窗口识别很快，但复杂网页、IDE、重叠窗口识别很差。
- 桌宠窗口或对话框被误当成用户正在看的目标窗口。

可能原因：

- 截图缩放后小字不可读。
- UI 层级复杂，VLM 优先描述布局而不是识别具体内容。
- 目标窗口被桌宠或对话框遮挡。
- 没有基于问题选择局部区域，整图输入导致细节稀释。

文章要点：

- 多模态模型的“视觉理解”同时包含 OCR、布局理解和语义理解，但这些能力都会受分辨率、裁剪、遮挡和 prompt 影响。
- 如果要回答细节，系统应先选图、选区域、再细看。
- 不应该用 OCR fallback 掩盖主链路问题；主链路不行就应该暴露并修复。

### 3.3 上下文断线：对话历史不是字符串拼接

典型失败：

```text
用户：帮我看看我继续问什么比较好
助手：要不要我帮你分析当前任务进展或下一步计划？
用户：可以
助手：你可以告诉我，我帮你分析当前任务进展或下一步计划吗？
```

问题本质：

- “可以”不是一个完整任务，它依赖上一轮助手的问题。
- 如果上一轮回答没有稳定进入历史，或者 planner 只看到当前问题，就会把“可以”当成孤立输入。
- 对话连续性不能只靠模型自己猜，系统要显式保留 turn state。

文章要点：

- 当前修复思路是增加 dialogue bridge：当用户短回复时，系统把上一轮助手提议作为承接提示注入 planner 和 answer。
- 这不是硬编码回答，而是恢复对话状态。
- 好的上下文管理要保护“用户正在回应什么”。

### 3.4 观察线和对话线混在一起

早期系统试图一边自动观察屏幕，一边回答用户问题。

问题：

- 用户打开对话框后，自动观察可能把对话框本身当成目标窗口。
- 回答过程中窗口变化会导致上下文漂移。
- 观察摘要和用户问答混在一起，模型无法区分“后台感知”和“当前对话目标”。

文章要点：

- 自动观察线：负责后台建立屏幕索引，不直接打扰用户。
- 用户对话线：负责承接意图、调用工具、回答问题。
- 用户点击观察或明确要求看屏幕时，才把视觉证据带入对话。

### 3.5 记忆不是数据库，也不是日志

项目曾经讨论过 Redis、PostgreSQL、Docker，后来决定拆掉。

原因：

- 桌宠主打轻量，即插即用，依赖 Docker 会破坏体验。
- 当前阶段需要的是本地短期记忆和 profile，而不是分布式缓存或团队数据库。
- Redis/PostgreSQL 解决的是部署和扩展问题，不解决“该记什么、何时取出、如何注入”的核心问题。

文章要点：

- 记忆管理的核心不是存储介质，而是写入策略、检索策略和注入策略。
- SQLite / 本地文件 / RuntimeStore 足够支撑原型。
- 真正重要的是区分：
  - profile：用户长期偏好和助手人设。
  - working memory：最近屏幕和任务状态。
  - conversation memory：当前会话历史。
  - episodic memory：重要事件或项目决策。

### 3.6 工具层不能变成关键词路由

原型一度倾向于：

```text
如果用户说“看 / 页面 / 截图” -> 走视觉
如果用户说“记住” -> 写记忆
否则 -> 普通聊天
```

问题：

- 关键词越补越多，系统越脆。
- 小模型失去判断空间，用户也会觉得对话死板。
- 错误不会暴露在工具调用日志里，而是表现为奇怪回答。

调整方向：

```text
用户问题
-> Tool Planner 选择少量工具
-> 工具执行并返回证据
-> Answer Model 基于证据自然回答
```

工具不宜过多。当前更适合给小模型暴露少量稳定工具：

- `screen.look(question)`：看当前屏幕或用户图片。
- `memory.search(query)`：查 profile、历史对话、近期目标。
- `memory.remember(note)`：仅用户明确要求时写入。
- `none`：普通陪伴和讨论。

文章要点：

- Hermes-like 的启发不是“工具越多越 Agent”，而是“工具边界要清楚、可审计、可调试”。
- 后端 provider 可以复杂，但模型可见工具要少。
- 工具调用日志应该暴露给高级调试，否则系统会越来越像黑盒。

### 3.7 KV cache 与多模态链路的冲突

讨论过的问题：

- 如果模型同时负责识图和对话，动态图片输入会破坏可缓存 prefix。
- 如果每轮都把窗口摘要、profile、记忆塞进 system prompt，也会让 prefix 不稳定。

当前理解：

- 稳定 system prompt 应只放协议、边界、回答原则。
- 用户可编辑 profile 单独作为低频上下文。
- 当前屏幕、历史截图、记忆检索结果作为动态 context packet 或工具结果。
- 视觉分析和对话回答可以分两层：平时观察建立索引，用户提问时按需细看。

文章要点：

- KV cache 不是记忆，只是性能优化。
- 为了缓存，系统提示词要稳定。
- 为了回答质量，动态上下文要被分层，而不是揉成一段大 prompt。

## 4. 推荐文章结构

### 开头：一个看起来很蠢的对话

用短例子开篇：

```text
用户：可以
助手：你可以告诉我你的具体需求吗？
```

引出问题：这不是单纯模型笨，而是系统没有告诉模型“用户在回应上一轮什么”。

### 第一部分：我以为截图 + 摘要就够了

讲原型初衷：

- 桌面悬浮机器人。
- 实时读取窗口。
- 生成摘要。
- 用户提问时基于摘要回答。

然后承认失败：

- 摘要能证明模型接入成功，但不能证明助手理解用户。
- 越强调摘要，越像机械屏幕分析器。

### 第二部分：真正的问题是上下文选择

拆四类上下文：

- 当前视觉证据：截图、局部图、窗口 metadata。
- 会话上下文：刚才说了什么、用户是在回应哪个问题。
- 任务上下文：用户最近在推进什么目标、卡在哪里。
- 人格与偏好：用户喜欢什么风格、不喜欢什么打扰。

强调：上下文管理不是“多塞”，而是“选对”。

### 第三部分：记忆管理不是加数据库

讲 Redis/PostgreSQL/Docker 的取舍：

- 它们不是坏技术，只是不适合轻量桌宠第一阶段。
- 对个人桌面伙伴来说，最重要的是本地、可解释、可迁移。
- SQLite + markdown profile + runtime store 更符合当前目标。

### 第四部分：工具层应该让模型自由，但边界清楚

讲 Hermes-like tool layer：

- 不再用 Python 关键词表决定一切。
- 让 LLM 选择 `screen.look / memory.search / memory.remember / none`。
- 工具结果进入最终回答，让模型自然组织语言。

注意：这里不要夸大为完整自主 Agent。它仍然是陪伴式桌面伙伴，不自动操作电脑。

### 第五部分：产品方向从摘要转向陪伴

讲最终认知变化：

- 用户不需要一个一直汇报屏幕的东西。
- 用户需要一个安静存在、在被叫到时能认真回应的小伙伴。
- 视觉、记忆、上下文都是后台器官，不是产品本体。

### 结尾：失败给出的架构原则

总结 5 条：

1. 摘要是索引，不是答案。
2. 截图是证据，不是上下文全部。
3. 记忆是选择，不是日志。
4. 工具是协议，不是关键词分支。
5. 陪伴式 Agent 的价值在于“接住用户”，不是“炫耀能力”。

## 5. 可引用的项目事实

当前项目事实可以写进文章：

- 项目是本地 Windows 桌面悬浮伙伴原型。
- 技术链路包含 FastAPI 后端、桌面悬浮窗、本地 VLM、SQLite RuntimeStore、markdown profile。
- 已移除 Redis / PostgreSQL / Docker 方向，转向轻量本地部署。
- 目前有自动观察线和用户对话线的拆分。
- 已尝试 Hermes-like 工具层：`screen.look`、`memory.search`、`memory.remember`、`none`。
- 已加入上下文使用率和交互轨迹调试思路。
- 最近修复过短回复对话断线问题：通过 dialogue bridge 把“可以/继续/好的”承接到上一轮助手提议。

不要写成：

- “已经实现了成熟 Agent”。
- “视觉识别已经很好”。
- “记忆管理已经完善”。
- “可以自动理解和操作用户电脑”。

更准确的表述：

- “这是一个正在逼近正确问题的原型。”
- “很多失败暴露了桌面 AI 伙伴最难的工程部分。”
- “目前的价值在于形成了清晰的架构判断。”

## 8. 文章验收标准

文章合格标准：

- 读者能理解为什么 VLM 看图不等于上下文管理完成。
- 读者能理解为什么 Redis/PostgreSQL 不是当前重点。
- 读者能理解为什么“摘要”会让桌宠显得机械。
- 读者能理解 memory 的核心是选择和注入，不是保存一切。
- 读者能看到一个从失败到架构原则的思考过程。

文章不合格信号：

- 只罗列技术栈。
- 过度赞美项目。
- 把问题归咎于“小模型太笨”。
- 用“未来可期”替代具体问题分析。
- 继续把摘要当作产品主角。
